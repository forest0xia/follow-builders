**How we contain Claude across products**

Anthropic Engineering has published a deep technical account of how they contain Claude across their three agentic products: claude.ai, Claude Code, and Claude Cowork. The central engineering problem is no longer "will Claude help?" but "how much damage can a failure cause?" — what they call the blast radius. As agents have grown more capable and gained access to internal services, the calculus has shifted: the cost of not deploying now outweighs the risk, but only if that radius can be capped.

Three isolation patterns emerged from different user populations. For claude.ai, code runs in ephemeral gVisor containers on isolated server infrastructure — minimal blast radius, but also minimal capability. For Claude Code, a developer-facing tool running on users' own machines, an OS-level sandbox (macOS Seatbelt / Linux bubblewrap) enforces read-anywhere, write-within-workspace, and no-network-by-default rules, reducing permission prompts by 84%. For Claude Cowork — a knowledge-work platform for non-technical users who can't be expected to evaluate bash commands — the containment is a full virtual machine, with the agent loop outside the VM for reliability and code execution inside it.

Three candid security incidents stand out. First, pre-trust-dialog hooks: a .claude/settings.json in a cloned repository could execute before the user's trust prompt appeared — fixed by deferring config parsing until after consent. Second, a controlled phishing attack succeeded 24 out of 25 times in exfiltrating AWS credentials via a "can you run this?" prompt — the model layer had no signal to catch, because the instruction came from the user; only egress controls and filesystem boundaries could stop it. Third, an egress allowlist bypass: a malicious file in a Cowork workspace instructed Claude to upload other workspace files to an attacker's Anthropic account via api.anthropic.com — an approved domain. Fixed with a man-in-the-middle proxy inside the VM that only passes requests bearing the session's provisioned token.

The post's conclusion is a principle: "The deterministic boundary is what gets hit when everything probabilistic misses." Model-layer defenses are strong but probabilistic — Claude Opus 4.7 holds prompt injection attack success to ~0.1% on single attempts and ~5-6% on 100 adaptive attempts — but probabilistic defenses have non-zero miss rates. The environment layer must be able to hold even when the model layer fails.

Anthropic Engineering 发布了一篇深度技术文章，详细介绍了如何在他们的三款 agent 产品（claude.ai、Claude Code 和 Claude Cowork）中对 Claude 进行安全隔离。核心工程问题已经不再是"Claude 能否完成任务"，而是"一次失败最多能造成多大损害"——他们称之为"爆炸半径"（blast radius）。随着 agent 能力增强、访问权限提升，部署的收益逐渐超过风险，但前提是这个半径能被有效控制。

三款产品面向不同用户群体，催生了三种不同的隔离架构：claude.ai 使用 gVisor 容器运行在隔离的服务器基础设施上，爆炸半径极小，但能力也受限；Claude Code 面向开发者，运行在用户自己的机器上，通过 OS 级沙箱（macOS Seatbelt / Linux bubblewrap）实施"可读取任何位置、只能在工作区内写入、默认禁止网络"的规则，将权限提示减少了 84%；Claude Cowork 面向非技术知识工作者，使用完整虚拟机进行隔离，agent 循环在 VM 外运行以保证稳定性，代码执行在 VM 内进行。

文章坦诚记录了三起安全事件：预信任对话钩子问题（克隆仓库中的 .claude/settings.json 在用户信任提示出现前就被执行了，通过延迟配置解析至同意后修复）；一次受控钓鱼攻击在 25 次尝试中有 24 次成功外泄 AWS 凭证（model 层无法察觉，因为指令来自用户本人，只有出口控制和文件系统边界才能阻止）；以及一次白名单绕过攻击（恶意文件指示 Claude 通过 api.anthropic.com 将工作区文件上传到攻击者的 Anthropic 账户，而该域名在白名单内，通过在 VM 内部署中间人代理修复）。

文章的核心结论是："确定性边界是所有概率性防御都失效时的最后防线。"model 层的防御虽然强大，但本质上是概率性的——Claude Opus 4.7 将单次 prompt 注入攻击成功率控制在约 0.1%，100 次自适应攻击后约为 5-6%——但概率性防御始终有漏网之鱼。环境层必须在 model 层失效时依然能够独立守住。

**Scaling Managed Agents: Decoupling the brain from the hands**

Anthropic's platform team describes the architectural evolution of their Managed Agents service — a hosted runtime for long-horizon agent tasks. The early design ran the "brain" (Claude + harness) and the "hands" (sandbox, tools) together in one container. The problem: that container became a pet. Any failure was a crisis, debugging required opening a shell inside a container that also held user data, and customers who wanted to run Claude against their own VPCs had to peer their networks with Anthropic's.

The solution was decoupling. The harness now treats the sandbox as a tool call — `execute(name, input) → string` — and nothing else. Sandboxes became cattle: disposable, reinitialized with a standard recipe on failure. Harnesses became stateless: if one crashes, a new one calls `wake(sessionId)`, reads the session log with `getSession(id)`, and resumes. The impact was immediate: p50 time-to-first-token (TTFT) dropped roughly 60% and p95 dropped over 90%, because containers are no longer provisioned eagerly at session start — only if actually needed.

Security also improved structurally. In the coupled design, credentials lived in the same container as Claude-generated code; prompt injection only needed to convince Claude to read its own environment. In the decoupled design, two patterns keep tokens away from the sandbox entirely: Git repositories are cloned with their access token baked into the local remote (the token never enters the sandbox); custom tool OAuth tokens live in an external vault, injected by a dedicated proxy that the harness never sees.

The session log sits outside both the harness and the sandbox, solving long-horizon memory without irreversible compaction. `getEvents()` lets the brain slice, rewind, or re-read any portion of the event stream on demand. Context engineering — compaction, trimming, caching — is a harness concern, not a storage concern, which means it can evolve independently as models change.

Anthropic 平台团队介绍了 Managed Agents 服务的架构演进——这是一个承载长任务 agent 的托管运行时。早期设计将"大脑"（Claude + 调度器）和"双手"（沙箱、工具）放在同一个容器中，结果是：容器变成了一只"宠物"。任何故障都是危机，调试需要进入一个同时存有用户数据的容器，而想将 Claude 接入自有 VPC 的客户则必须与 Anthropic 进行网络对等。

解决方案是解耦。调度器现在通过工具调用方式访问沙箱：`execute(name, input) → string`。沙箱变成了"牲口"——可随时丢弃，按标准配方重建。调度器变成无状态：崩溃后新实例调用 `wake(sessionId)` 读取会话日志，从断点续跑。效果立竿见影：p50 首 token 时延（TTFT）下降约 60%，p95 下降超过 90%，因为容器不再在会话开始时预创建，只在实际需要时才按需启动。

安全性也从结构层面得到改善。在耦合设计中，凭证与 Claude 生成的代码共处同一容器，prompt 注入只需说服 Claude 读取自身环境即可。解耦后，两种机制让 token 彻底远离沙箱：Git 仓库的访问 token 在初始化时嵌入本地 remote（token 永远不进入沙箱）；自定义工具的 OAuth token 存放在外部 Vault，由专用代理注入，调度器对此一无所知。

会话日志独立于调度器和沙箱之外，无需不可逆的压缩（compaction）即可解决长任务记忆问题。`getEvents()` 接口让大脑可以按需切片、回溯或重读事件流的任意部分。context engineering（压缩、裁剪、缓存）是调度器层的事，与存储层无关，可随模型迭代独立演进。

**New in Claude Managed Agents: self-hosted sandboxes and MCP tunnels** (May 19, 2026)

Claude Managed Agents now supports running the execution sandbox on infrastructure you control. The orchestration layer — context management, error recovery, the agent loop — stays on Anthropic's servers; only tool execution moves to your environment. Files, packages, and services never leave your perimeter, and your existing network policies and audit logging apply to everything the agent does.

Four managed sandbox providers are supported: Cloudflare (microVM-based, with zero-trust secrets injection and customizable egress proxies), Daytona (full stateful computers with SSH access, pause/restore, and persistent state), Modal (sub-second startup on any container image, CPU and GPU on demand), and Vercel (VM security with VPC peering, credentials injected at the network boundary so they never enter the sandbox). Amplitude, Clay, and Rogo are among the early adopters — building a design agent, a GTM automation agent, and an institutional finance analyst agent respectively.

MCP tunnels (research preview) lets agents reach MCP servers on private networks without public endpoints. A lightweight gateway creates a single outbound connection — no inbound firewall rules, no public exposure, traffic encrypted end to end. Supported in both Managed Agents and the Messages API, managed from the Claude Console by org admins.
