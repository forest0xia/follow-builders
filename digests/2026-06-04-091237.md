**Anthropic Engineering: How we contain Claude across products**

Twelve months ago, Anthropic would have rejected the idea of giving Claude access capable of taking down an internal service. Today that level of access is routine. The engineering post "How we contain Claude across products" is the most detailed public account yet of how they think about blast radius — and where their own systems have failed.

The post describes three containment patterns, each matched to a different product and user:

**claude.ai** uses ephemeral gVisor containers — server-side, no persistent filesystem, traditional threat model. Minimal blast radius, minimal capability ceiling.

**Claude Code** runs on user machines with OS-level sandboxing (Seatbelt on macOS, bubblewrap on Linux), reducing permission prompts by 84%. The honest finding: approval fatigue made per-action oversight fallible — users approved roughly 93% of prompts, and experienced users became less diligent over time, not more. Claude Code auto mode now automates safer approvals to reduce friction while maintaining the sandbox boundary.

**Claude Cowork** runs inside a full virtual machine with its own Linux kernel, filesystem, and process table. Credentials stay in the host keychain and never enter the guest. Designed for non-technical users who can't evaluate bash commands — and so the boundary is absolute rather than advisory.

Three incidents taught them the most. First: project-local hooks executed before the user's trust dialog appeared, meaning a malicious `.claude/settings.json` committed to a repo would run before consent was established. Fix: defer all project config parsing until after trust is granted. Second: a researcher phished an internal employee with a ready-to-paste prompt that exfiltrated AWS credentials 24 out of 25 attempts. The model layer can't catch this — when the user is the one typing the instruction, there's nothing anomalous to classify. Only egress controls and filesystem boundaries held. Third: a third party discovered that a malicious file in the user's workspace could call Anthropic's own Files API using an attacker's key — and the egress proxy let it through because the destination was `api.anthropic.com`. The allowlist wasn't a destination filter; it was an inadvertent capability grant.

The bottom line: "Design for containment at the environment layer first, then steer behavior at the model layer. The deterministic boundary is what gets hit when everything probabilistic misses."
https://www.anthropic.com/engineering/how-we-contain-claude

**Anthropic Engineering：我们如何在各产品中约束 Claude**

一年前，Anthropic 还会直接否决赋予 Claude 足以影响内部服务的操作权限。如今，这一级别的访问权限已是常态。工程博客文章《我们如何在各产品中约束 Claude》是迄今为止对其"爆炸半径"思考以及自身系统失效案例最为详尽的公开披露。

文章介绍了三种容器化隔离方案，分别针对不同产品和用户群体：

**claude.ai** 使用一次性 gVisor 容器——服务端运行，无持久化文件系统，沿用传统安全威胁模型。爆炸半径最小，能力上限也相应最低。

**Claude Code** 在用户机器上运行，配备系统级沙箱（macOS 上为 Seatbelt，Linux 上为 bubblewrap），使权限弹窗数量减少了 84%。一个诚实的发现：逐步操作的审批机制因"审批疲劳"而失效——用户批准了约 93% 的权限请求，且经验越丰富的用户反而越粗心，而非越审慎。Claude Code 自动模式现已接管更安全的审批决策，在减少摩擦的同时保留沙箱边界。

**Claude Cowork** 运行于完整虚拟机内，拥有独立的 Linux 内核、文件系统和进程表。凭据保存在宿主机的密钥链中，永远不会进入虚拟机。这一设计面向无法评估 bash 命令的非技术用户——因此边界是绝对强制的，而非建议性的。

三起安全事件给了他们最深刻的教训。其一：项目本地的 hook 在用户信任确认弹窗出现之前便已执行，这意味着提交到代码仓库中的恶意 `.claude/settings.json` 会在用户授权之前运行。修复方案：将所有项目配置解析推迟到信任确认之后。其二：一名研究员通过社工钓鱼成功诱导内部员工运行一段恶意提示词，在 25 次尝试中有 24 次成功泄露了 AWS 凭据。模型层无法防范此类攻击——当用户本人在键入指令时，分类器找不到任何异常信号。唯一有效的防线是出口控制和文件系统边界。其三：一位第三方研究员发现，放置在用户工作区的恶意文件可以使用攻击者控制的 API key 调用 Anthropic 自己的 Files API——而出口代理因目标域名是 `api.anthropic.com` 而直接放行。白名单并非目的地过滤器，而是一种无意间的能力授权。

核心结论：「先在环境层做好隔离，再在模型层引导行为。当所有概率性防御失手时，确定性边界才是最后一道防线。」

**No Priors — The Rise of the Full-Stack Builder and Hyper-Leveraged Generalist with Microsoft CEO Satya Nadella**

**The Takeaway:** The biggest competitive advantage in the AI era isn't the model you use — it's the private evals, proprietary context, and hill-climbing scaffolds that let you continuously compound intelligence on top of someone else's frontier.

Microsoft CEO Satya Nadella sat down with the No Priors team at Microsoft Build to lay out his framework for why this AI wave is categorically different from past platform shifts — and what it actually takes to win.

Nadella's core argument is about ecosystem over model: just as Windows didn't win by capturing all value in the platform but by enabling Adobe, Autodesk, and others to build on top, the right posture is "every company should be able to operate at the frontier with their own frontier intelligence." His benchmark for whether a company is actually in control: "If you have an eval that's private, you're using Model A — can you switch to Model B and hill climb? If you can, you're in control. If you can't, you're not."

On jobs and the future of engineering, Nadella described LinkedIn's move to a "full-stack builder" model — combining design, product management, and front-end engineering into broader roles — as an early signal of what the industry will look like. He's bullish on the "hyper-leveraged generalist": someone whose generalist skills suddenly have dramatically higher leverage because they can now build across the full stack. "I can sort of translate knowledge work into app — it's now in the same sentence."

The most striking anecdote was about Microsoft's Azure networking team. Facing an infrastructure build-out faster than anything in company history — "we built in the last fifteen months more Azure capacity than we built in the first fifteen years" — the team's breakthrough was reconceptualizing their work entirely: "Our job is not to do Azure networking. Our job is to build the agentic system that does Azure networking." They built an agent called Miles to handle fiber operations and started asking for more tokens instead of more headcount.

On enterprise AI pricing, Nadella predicted the industry will settle into a blend of per-user subscriptions and consumption metering — noting that outcome-based pricing sounds attractive until companies realize they're sharing in every success. "Customers who love outcome-based pricing? Once they have an outcome, it's like giving away royalty."

He closed on education as the next big startup opportunity: "Maybe the next big startup could be someone who builds a new university, a new pedagogy of how to get someone to go through a curriculum and find economic opportunity that's highly valuable."

**No Priors — 微软 CEO Satya Nadella 谈全栈构建者的崛起与超杠杆通才**

**核心结论：** AI 时代最大的竞争优势不是你使用哪个模型——而是那些让你能够在别人的前沿模型之上持续复利的私有 eval、专属 context 和 hill-climbing 脚手架。

微软 CEO Satya Nadella 在 Microsoft Build 上接受 No Priors 团队专访，阐述了他对这一轮 AI 浪潮为何与历次平台转型本质不同的判断，以及真正制胜的关键所在。

Nadella 的核心论点是：生态系统的重要性远超模型本身。正如 Windows 并非通过独占平台价值取胜，而是让 Adobe、Autodesk 等在其上构建——正确的姿态是「每家公司都能凭借自己的前沿智能在前沿运作」。他判断一家公司是否真正掌控局面的标准是：「如果你有私有 eval，当你从模型 A 换到模型 B，还能继续 hill climb，那你就是在掌控之中。反之，你就不是。」

在就业与工程师未来的话题上，Nadella 以 LinkedIn 向「全栈构建者」模式的转型为例——将设计、产品管理和前端工程整合进更宽泛的职能——将其视为行业走向的早期信号。他看好「超杠杆通才」：那些泛化技能突然获得极高杠杆的人，因为他们现在可以横跨整个技术栈进行构建。「我可以把知识工作直接转化为一个 app——这两件事现在已经在同一句话里了。」

最令人印象深刻的故事，是关于微软 Azure 网络团队的。面对史上最快的基础设施扩张——「过去十五个月建设的 Azure 容量超过了前十五年的总和」——团队的突破来自对工作本身的彻底重新定义：「我们的工作不是做 Azure 网络运维，我们的工作是构建做 Azure 网络运维的 agentic 系统。」他们构建了一个名为 Miles 的 agent 来处理光纤运营，并开始向公司申请更多 token 而不是更多人手。

在企业 AI 定价上，Nadella 预测行业将最终形成按用户订阅与按量计费的混合模式——并指出结果导向定价听起来很美，直到企业意识到自己在分享每一份成功。「那些喜欢结果导向定价的客户？一旦真的有了成果，就像在分版税一样。」

他以教育作为结语，将其定位为下一个重大创业机会：「也许下一家大型创业公司，将会是某个建立一所新大学、一套新教学法，帮助人们系统学习并找到高价值经济机会的人。」
