**Anthropic Engineering: An update on recent Claude Code quality reports**

Anthropic published a postmortem on three separate bugs that degraded Claude Code, the Claude Agent SDK, and Claude Cowork between March and April 2026 — all now fixed as of April 20 (v2.1.116).

The three issues: (1) On March 4, Claude Code's default reasoning effort was quietly downgraded from "high" to "medium" to reduce UI freeze times — a tradeoff users hated. Reverted April 7. (2) On March 26, a change cleared Claude's older thinking from its context during long agentic tasks, breaking continuity. (3) A third issue affected the Claude Agent SDK's tool-calling behavior in ways that made responses feel degraded. The API was unaffected throughout.

What changes going forward: Anthropic committed to better internal eval coverage before shipping behavior-touching changes, and more explicit communication to users when defaults are intentionally shifted. The post is a model of transparent incident reporting: specific dates, specific version numbers, and direct acknowledgment that they made wrong tradeoffs.

https://www.anthropic.com/engineering/april-23-postmortem

Anthropic Engineering 发布了一份关于 2026 年 3 月至 4 月间三个独立 bug 的事后复盘，这些 bug 导致 Claude Code、Claude Agent SDK 和 Claude Cowork 表现下降，均已于 4 月 20 日（v2.1.116）修复完毕。

三个问题分别是：(1) 3 月 4 日，Claude Code 的默认推理强度被悄悄从"高"降为"中"，以减少 UI 冻结时间——但用户对这个取舍并不买账，已于 4 月 7 日回滚。(2) 3 月 26 日，一个变更在长 agent 任务中清除了 Claude 的早期思考记录，导致上下文连贯性断裂。(3) 第三个问题影响了 Claude Agent SDK 的工具调用行为，使响应质量感知明显下降。整个过程中 API 层未受影响。

**Claude Blog: Built-in memory for Claude Managed Agents**

Anthropic is shipping persistent memory for Claude Managed Agents, available now in public beta. Agents can now learn across sessions — not via a proprietary database, but through a filesystem-based memory layer that stores memories as plain files. Developers retain full control: memories can be exported, managed via API, and inspected or deleted at will.

The architecture choice is deliberate. Because Claude is already strong at file operations (bash, code execution), mounting memory directly onto a filesystem means agents use the same tools they already rely on rather than a separate memory abstraction. The system is optimized for long-running agents that need to share what they've learned with each other — not just remember things within a single session. Enterprise deployments get scoped permissions and audit logs.

https://claude.com/blog/claude-managed-agents-memory

Anthropic 为 Claude Managed Agents 推出了持久化记忆功能，现已进入公开 beta。Agent 现在可以跨会话持续学习——但不是通过专有数据库，而是通过一个基于文件系统的记忆层，将记忆内容存储为普通文件。开发者拥有完全控制权：记忆可以导出、通过 API 管理，并可随时查看或删除。

这个架构选择是有意为之的。由于 Claude 本身在文件操作（bash、代码执行）方面已经很强，将记忆直接挂载到文件系统上意味着 agent 继续使用它已经熟悉的工具，而不是引入一套额外的记忆抽象层。该系统专门针对长时间运行的 agent 进行了优化，支持多个 agent 之间共享所学内容，而不只是在单个会话内记住事情。企业级部署具备权限隔离和审计日志。

**AI & I by Every: The Secrets of Claude's Platform From the Team Who Built It**

The Takeaway: The model and the harness are now inseparable — generic, model-agnostic architectures are a relic of an earlier, slower era of AI development.

Angela (Head of Product, Claude Platform) and Caitlin (Head of Engineering, Claude Platform) at Anthropic sat down with the AI & I podcast to pull back the curtain on how Claude Managed Agents was actually built, and why the platform has evolved the way it has.

The most counterintuitive thing they said: Anthropic built Claude Managed Agents primarily because Anthropic itself got tired of rebuilding the same cloud agent infrastructure over and over internally. "We're done building this for ourselves. We're doing it once in a way that's gonna really work from everything that we've learned, but also for all the people who are doing it." That's a different origin story than "we saw market demand."

On the fear of model lock-in — a concern every team building with AI has right now — Angela had a sharp reframe: the idea that you can maintain a generic harness and hot-swap any model was "mildly reasonable" a couple of years ago when models across labs were more similar. Now, every lab is pushing in different directions, and the harness and the model "get very paired." You still want redundancy, but the abstraction layer is moving up, not disappearing.

On the future of the platform: the vision is that Claude eventually gets good enough at understanding itself that it can figure out which model to use, spin up the right sub-agents, and effectively write its own architecture on the fly. The platform's job then becomes scaling infrastructure fast enough to support that.

The practical tension at the center of the conversation: teams like Every are running agents on Mac minis with thousand-line Python files, close to the edge of what's possible, and they're worried that adopting Claude Managed Agents means trading flexibility for comfort. Angela and Caitlin's answer was essentially: first-party Anthropic products run on the same platform infrastructure, so the feature gap should close over time.

https://www.youtube.com/watch?v=lLypHkIVLqc

**AI & I by Every《Claude 平台背后的秘密，来自亲手构建它的团队》**

核心洞见：模型和 harness 现在已经不可分割——通用的、模型无关的 agent 架构，是 AI 发展更早、更慢阶段的遗产。

Anthropic 的 Angela（Claude 平台产品负责人）和 Caitlin（Claude 平台工程负责人）做客 AI & I 播客，揭秘了 Claude Managed Agents 的真实构建历程，以及平台为何走到今天这条路。

他们说的最让人意外的事：Anthropic 之所以构建 Claude Managed Agents，主要是因为 Anthropic 自己受够了反复在内部重建相同的云端 agent 基础设施。"我们不想再为自己重复造轮子了。这次我们要做一个真正靠谱的版本，把我们学到的一切都用上，也让所有正在做同样事情的人受益。" 这和"我们看到了市场需求"是完全不同的起源故事。

关于模型锁定的顾虑——这是每个正在使用 AI 构建产品的团队都有的担忧——Angela 给出了一个犀利的重新框架：过去认为可以用通用 harness 热换任意模型的想法，在两三年前各家模型差异不大时"勉强说得通"。而现在，每家实验室正在向不同方向推进，harness 和模型"越来越紧密耦合"。你仍然需要冗余备份，但抽象层在往上走，而不是消失。

关于平台的未来：愿景是 Claude 最终对自身的理解强大到可以自己判断该用哪个模型、启动哪些子 agent，实际上在运行时动态生成自己的架构。届时，平台的任务就变成了：以足够快的速度扩展基础设施来支撑这一切。

对话中贯穿始终的核心张力是：像 Every 这样的团队，正在用 Mac mini 和几千行 Python 文件跑 agent，始终保持在技术前沿，他们担心采用 Claude Managed Agents 意味着用灵活性换舒适度。Angela 和 Caitlin 的回答本质上是：Anthropic 的自有产品就跑在同一套平台基础设施上，所以功能差距随着时间推移应该会收窄。
