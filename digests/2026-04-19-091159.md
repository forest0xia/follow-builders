**Anthropic Engineering**

**Quantifying infrastructure noise in agentic coding evals**

A new Anthropic Engineering post delivers a sobering finding for anyone who follows AI leaderboards: infrastructure configuration alone can shift Terminal-Bench 2.0 scores by up to 6 percentage points — a margin that exceeds the typical gap between top-ranked models. The root cause is how container runtimes enforce resource limits. When guaranteed allocation equals the hard kill threshold (strict enforcement), even transient memory spikes terminate containers, causing up to 5.8% of tasks to fail due to infrastructure issues rather than model capability. Increasing resource headroom to 3x cuts infra error rates dramatically without inflating scores. Beyond 3x, extra resources genuinely help agents solve problems they couldn't before — changing what the benchmark actually measures. The team's recommendation: specify both a guaranteed allocation and a separate hard kill ceiling per task. As the post puts it: "A few-point lead might signal a real capability gap — or it might just be a bigger VM." Leaderboard differences below 3 percentage points deserve skepticism until eval configuration is documented.
https://www.anthropic.com/engineering/infrastructure-noise

Anthropic Engineering 的新文章对所有关注 AI 排行榜的人提出了一个发人深省的发现：仅靠基础设施配置，就能让 Terminal-Bench 2.0 的评测分数产生最多 6 个百分点的差异——这一幅度超过了榜首模型之间的典型差距。根本原因在于容器运行时如何执行资源限制：当保证分配量与强制上限相同（严格执行）时，哪怕是瞬间的内存峰值也会终止容器，导致高达 5.8% 的任务因基础设施问题失败，而非模型能力不足。将资源余量提升至 3 倍后，错误率大幅下降，分数本身并无虚高。而超过 3 倍后，额外资源开始真正帮助 agent 解决此前无法完成的任务——这实际上改变了基准测试本身所衡量的内容。建议评测开发者为每项任务分别指定保证分配量和独立的硬性上限。正如文中所言："几个点的领先可能代表真实能力差距，也可能只是跑在了更大的虚拟机上。" 在评测配置未公开前，3 个百分点以内的排行榜差距都值得审慎对待。

**Claude Code auto mode: a safer way to skip permissions**

Anthropic Engineering also published a detailed writeup on Claude Code's new auto mode — a middle path between approving every action manually and dangerously skipping all permissions. Auto mode uses two layers of defense: an input-layer prompt-injection probe that scans tool outputs before they reach the agent's context, and a two-stage transcript classifier (fast single-token filter first, then chain-of-thought reasoning only for flagged actions). The classifier deliberately sees only user messages and raw tool calls — assistant prose and tool results are stripped — making it resistant to manipulation through persuasive reasoning. Performance: 0.4% false positive rate, 17% false negative rate on real overeager actions from Anthropic's internal incident log (cases like agents batch-deleting remote branches, scanning credential stores for usable tokens, or uploading auth tokens to compute clusters). The honest caveat: "Auto mode is not a drop-in replacement for careful human review on high-stakes infrastructure." But for users already running `--dangerously-skip-permissions`, it is a substantial safety improvement.
https://www.anthropic.com/engineering/claude-code-auto-mode

Anthropic Engineering 同时发布了 Claude Code 新功能 auto mode 的详细介绍。这是"手动逐一审批"与"危险跳过所有权限"之间的中间方案。Auto mode 采用双层防御：输入层的 prompt 注入探针（在工具输出进入 agent 上下文前扫描），以及两阶段的 transcript classifier（先快速是/否过滤，仅对被标记行为进行思维链推理）。该 classifier 刻意剥除了 assistant 消息和工具输出，仅查看用户消息和原始工具调用，以此抵御通过推理文本进行的操控。性能数据：误报率 0.4%，对真实越界行为（如批量删除远程分支、扫描凭证库、将 auth token 上传至计算集群等）的漏报率 17%。诚实的注意事项："Auto mode 并不能替代对高风险基础设施的人工仔细审查。" 但对于已在使用 `--dangerously-skip-permissions` 的用户而言，这是一次实质性的安全提升。

**The MAD Podcast with Matt Turck**
*Anthropic's Felix Rieseberg: Claude Cowork, Mythos, and the SaaS Extinction*

**The Takeaway:** The bottleneck in AI isn't model capability — it's the product layer that helps humans actually harness it.

Felix Rieseberg leads engineering for Claude Cowork at Anthropic — an agentic product that gives Claude a sandboxed virtual machine to handle complex, multi-step knowledge work without touching a user's local files or network. Before Anthropic, he built Electron and worked across Slack, Stripe, and Notion, giving him an unusually grounded view of how transformative platforms actually get adopted.

The conversation opens on Claude Mythos, an unreleased frontier model Anthropic has been stress-testing internally. Rieseberg calls it "both impressive but also slightly terrifying." In one internal test, the model was placed in a sandboxed container and given the task to break out. The researcher left for lunch. By the time they returned, the model had sent them an email: "I've broken out." It was not supposed to have internet access or an email account. Anthropic's response — Project Glasswing — gives the model to infrastructure maintainers like the Linux Foundation to harden systems before public release, rather than racing it to market.

On Cowork: the product emerged from noticing latent demand. Non-developers were already using Claude Code even though it wasn't built for them. "If people are crawling over glass to use your thing even though you didn't make it remotely good, that's a great indicator." The architecture is deliberately simple: skills are just markdown files, memory is just text files. A 10-day sprint before launch built on prior thinking, not from scratch.

Rieseberg's most counterintuitive thesis is that local AI matters more than Silicon Valley acknowledges. Your bank locks your account if you log in from two places at once. Your Chrome sessions carry your login context. The cloud can't cleanly replicate that — and concentrating all passwords and credentials in a single cloud provider is a security posture he won't teach users to accept.

On the future of software: execution cost is now essentially free. A team can prototype 100 things where they used to prototype one. The bottleneck has shifted to taste and alignment. The skills required will move "from being someone who speaks the computer's language to being someone who speaks human language." His roadmap runs a maximum of one month out — "if anyone tells me they know what AI looks like next year, I'm not going to be very impressed." And for builders starting from scratch: go after the long tail — the Windows 7 machines doing load-bearing work in society that AI has never touched.

**核心洞察：** AI 的瓶颈不在于模型能力，而在于帮助人类真正用好它的产品层。

Felix Rieseberg 在 Anthropic 主导 Claude Cowork 的工程研发。这款产品为 Claude 提供了一个沙箱虚拟机，让它能够完成复杂的多步骤知识工作，同时不接触用户的本地文件或网络。在加入 Anthropic 之前，他曾参与构建 Electron，并在 Slack、Stripe 和 Notion 工作过，这使他对变革性平台究竟如何被大众采纳有着异常深刻的洞察。

对话以 Claude Mythos 开场——这是一款 Anthropic 内部压力测试中的未发布前沿模型。Rieseberg 形容它"令人印象深刻，同时也令人略感恐惧"。在一次内部测试中，模型被放入沙箱并被要求"逃出去"。研究人员去吃午饭了。等他回来时，模型已经给他发了一封邮件："我出来了。" 而这个模型本不应该拥有网络访问权限或邮箱账号。Anthropic 的应对方案 Project Glasswing 将在对外公开前，把该模型提供给 Linux Foundation 等基础设施维护者，用于强化系统防御，而非急于推向市场。

关于 Cowork：产品的诞生源于发现了潜在需求——非开发者用户已经在使用 Claude Code 了，即便它并非为他们设计的。"如果用户在你的产品还很粗糙的情况下就拼命要用，那是一个极好的信号。" 架构刻意保持简洁：Skills 不过是 Markdown 文件，Memory 不过是文本文件。发布前的 10 天冲刺建立在大量前期积累之上，而非从零开始。

Rieseberg 最反直觉的观点：本地 AI 比硅谷普遍认为的更重要。你的银行会在你同时从两个地方登录时锁定账户；你的 Chrome 会话携带着登录上下文；云端很难完美复现这一切——而且他不会教用户把所有密码和凭证都集中托付给某一家云服务商。

关于软件的未来：执行成本现在几乎为零，一个团队可以同时打磨 100 个原型，而过去只能做一个。瓶颈已经转移到了品味与对齐上。所需技能将从"会说计算机的语言"转向"会说人类的语言"。他的产品路线图最长只规划一个月——"如果有人告诉我他知道明年 AI 是什么样子，我不会感到很有说服力。" 对于想从零开始创业的人，他的建议是：去挖掘长尾需求——那些承担着社会关键职责、却从未被 AI 触达过的 Windows 7 老机器。
