### Anthropic Engineering

**An update on recent Claude Code quality reports**
https://www.anthropic.com/engineering/april-23-postmortem

Anthropic Engineering published a detailed postmortem on three separate issues that caused Claude Code quality degradation for some users over the past month. All three have been resolved as of April 20 (v2.1.116):

1. **Reasoning effort downgrade (March 4):** Claude Code's default reasoning effort was quietly changed from high to medium to reduce latency, resulting in noticeably worse output. Reverted April 7. All models now default to xhigh effort for Opus 4.7 and high for others.

2. **Caching bug that erased thinking history (March 26):** An optimization meant to clear old thinking blocks from idle sessions had a bug that wiped reasoning history on *every* turn for the rest of a session — making Claude seem forgetful and burning usage limits faster. Fixed April 10.

3. **Verbosity reduction prompt (April 16):** A system prompt instruction limiting responses to ≤25 words between tool calls and ≤100 word final responses hurt coding quality. Reverted April 20.

Because each change hit different users at different times, the aggregate looked like broad inconsistent degradation. As of April 23, Anthropic reset usage limits for all subscribers. Going forward, they're adding tighter eval coverage for every system prompt change and broader soak periods before rollout.

---

Anthropic Engineering 发布了一份详细的事后复盘报告，梳理了过去一个月导致部分用户 Claude Code 质量下降的三个独立问题。所有问题已于 4 月 20 日（v2.1.116）修复：

**Scaling Managed Agents: Decoupling the brain from the hands**
https://www.anthropic.com/engineering/managed-agents

Anthropic Engineering explained the architecture behind Managed Agents — a hosted service for long-horizon agent work. The key design insight: decouple the "brain" (Claude + harness) from the "hands" (sandbox/tools) and the "session" (event log), so each can fail or be replaced independently.

The original coupled design had serious problems: if a container failed, the session was lost; credentials were reachable from inside the sandbox (a security risk); and scaling required one container per brain, with each paying full provisioning latency upfront. The decoupled design solved all three. Containers become "cattle" — when one dies, the harness catches the error as a tool-call failure and a new one is provisioned on demand. The harness can be rebooted with just a session ID. Credentials are held outside the sandbox via git token injection or an OAuth vault. And provisioning containers only when needed dropped median time-to-first-token by ~60% and p95 by over 90%.

The session log also serves as durable context storage outside Claude's context window, enabling long-horizon tasks without irreversible compaction decisions.

---

Anthropic Engineering 详解了 Managed Agents 背后的架构——这是一项用于长时间 agent 任务的托管服务。核心设计理念：将"大脑"（Claude + harness）与"双手"（sandbox/工具）和"会话"（事件日志）解耦，使每个部分均可独立失败或替换。

原有耦合设计存在严重问题：容器失败则会话丢失；凭证在 sandbox 内可访问（安全隐患）；扩展时每个大脑需要一个容器，且每次启动都需付出完整的初始化延迟。解耦设计解决了所有这些问题。容器变成"无状态牲口"——失败时 harness 将其捕获为工具调用错误，并按需重新分配新容器。harness 只需一个 session ID 即可重启。凭证通过 git token 注入或 OAuth vault 存储在 sandbox 之外。按需分配容器将中位首 token 时间降低约 60%，p95 降低超过 90%。

会话日志还作为 Claude 上下文窗口之外的持久化上下文存储，支持长时间任务而无需做出不可逆的压缩决策。

## PODCASTS

**The MAD Podcast with Matt Turck — OpenAI's Compute Chief: We Can't Build Fast Enough | Sachin Katti**
https://www.youtube.com/watch?v=wEZBlmvxx4o

**The Takeaway:** OpenAI's compute demand is so insatiable that every time they've thought they had enough and slowed down, they've regretted it — and the physical world simply can't build fast enough to keep up.

Sachin Katti, Head of Industrial Compute at OpenAI, brings a rare inside view of what may be the largest infrastructure buildout in history. A Stanford CS professor-turned-startup-founder-turned-Intel CTO, Katti now oversees the full lifecycle of compute acquisition at OpenAI: land, power, data centers, chips, financing, and capacity allocation.

The numbers are staggering. OpenAI is spending roughly $50 billion on compute this year alone, in an industry-wide spend projected at $700 billion. But Katti's biggest worry isn't overbuilding — it's the opposite. "Anytime you have thought you have enough compute, we can slow down. Always negatively surprises... we should not have slowed down." The physical supply chain simply can't accelerate fast enough: gas turbines, transformer substations, and skilled electricians are all bottlenecked. OpenAI is now investing in grid infrastructure — solar, hydro, gas — wherever it builds data centers, committing to add generation capacity rather than just draw from existing grids.

On custom silicon: OpenAI designed its Jalapeno chip in just nine months (a speed Katti credits to their Broadcom partnership, a veteran TPU design team, and AI helping design the chip itself). "We do believe that the world of recursion is not that far where AI will design the systems it needs to train and run the next generation of AI." The strategic logic: knowing the exact inference workload enables co-designed hardware that maximizes tokens per watt — critical when compute is constrained by power.

On inference vs. training: the distinction is collapsing. Synthetic data generation, post-training, test-time compute — "when we say training, a lot of the compute actually is inference even in that phase."

On MRC, OpenAI's new networking protocol: a multipath packet-spraying approach that routes traffic across hundreds of paths simultaneously in a 100,000-GPU cluster, so single link failures never stop training runs.

The most counterintuitive claim: as AI does more AI research, the number of experiments that can be run is no longer bottlenecked by the number of human researchers. Compute demand from research will therefore explode in ways that human-limited labs never could.

---

**播客亮点：** OpenAI 对算力的需求如此旺盛，以至于每次他们认为已经够用、可以放缓时，结果总是后悔——而现实世界根本无法以足够快的速度完成建设。

Sachin Katti 是 OpenAI 工业算力负责人，为我们带来了这或许是人类史上最大基础设施建设项目的内部视角。他曾是斯坦福 CS 教授、连续创业者和英特尔 CTO，如今掌管 OpenAI 算力获取的全生命周期：土地、电力、数据中心、芯片、融资和容量分配。

数字令人咋舌。OpenAI 今年仅在算力上的支出就约达 500 亿美元，而整个行业的预计支出高达 7000 亿美元。但 Katti 最担忧的不是过度建设，而恰恰相反。"每次你以为算力够用了，可以放缓脚步，结果总是令人失望……我们不应该放缓的。" 现实世界的供应链根本跟不上：燃气轮机、变电站和熟练电工全都成了瓶颈。OpenAI 现在在每个新建数据中心的地区都投资电网基础设施——太阳能、水电、天然气——承诺增加发电容量而非仅仅从现有电网取电。

关于定制芯片：OpenAI 仅用九个月便完成了 Jalapeno 芯片的设计（Katti 将速度归功于与 Broadcom 的合作、经验丰富的 TPU 设计团队，以及 AI 参与芯片设计本身）。"我们相信，AI 设计它训练和运行下一代 AI 所需系统的'递归时刻'已经不远了。" 战略逻辑在于：精确知道推理工作负载，就能协同设计硬件，最大化每瓦 token 产出——在算力受制于电力的当下，这至关重要。
