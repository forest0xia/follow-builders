**Anthropic Engineering: How we contain Claude across products**

Anthropic Engineering published a deep-dive into how they cap the "blast radius" of Claude-powered agents across their products. The core finding is uncomfortable: supervision doesn't scale. Their telemetry showed users approved 93% of permission prompts, and attention degrades with each one. The more approvals a user sees, the less diligent they become. So Claude Code recently shipped an "auto mode" to reduce approval fatigue — but probabilistic defenses always have a non-zero miss rate.

The real engineering focus has shifted to containment: sandboxes, virtual machines, and egress controls that constrain what an agent is *able* to do rather than watching what it does. As the post frames it: "The risk-reward calculation tips heavily toward adoption, as long as products can be made safe. The engineering question becomes how to cap the blast radius."

The post also reveals that one model — Claude Mythos Preview — was held back in April 2026 specifically because its blast radius was deemed too high. Broader release is expected as defenders harden critical systems and safeguards mature. This is safety reasoning at the engineering layer, not the policy layer: practical decisions about what access an agent should never have.
https://www.anthropic.com/engineering/how-we-contain-claude

Anthropic Engineering 发布了一篇深度文章，详解他们如何为各产品中 Claude agent 的"爆炸半径"设置上限。核心发现令人不安：监督机制根本无法扩展。他们的遥测数据显示用户对权限提示的批准率高达93%，且随着审批次数增加注意力持续下降。为此，Claude Code 最近推出了"自动模式"来减轻审批疲劳——但概率性防御的漏检率永远不为零。

真正的工程重心已转向"隔离（containment）"：通过沙盒、虚拟机和出口流量控制来限制 agent 的能力边界，而非监控它做了什么。文章的表述非常直接："只要产品能做到安全，风险收益的算盘就会明显偏向部署。工程问题变成了：如何给爆炸半径设定上限。"

文章还披露，有一款模型——Claude Mythos Preview——在2026年4月被明确押后发布，原因正是潜在破坏范围被判定过高。随着关键系统防护成熟和安全措施完善，更广泛的发布预计将在适当时机推进。这是工程层面的安全推理，不是政策层面的原则声明：它讨论的是 agent 不应拥有哪些访问权限的实际决策。

**Training Data: How Cursor Trained Composer on Fireworks: Distributed Infrastructure for High-Performance RL**

**The Takeaway:** Training a model exclusively for one product — not coding in general, but software engineering *inside Cursor only* — turns out to be both cheaper and more capable than relying on frontier models, but requires reinforcement learning infrastructure so demanding that Cursor had to partner with Fireworks to build it.

Federico, research lead on Composer 2 at Cursor, makes the case with a simple analogy: think of a model as a storage drive with a fixed number of bits. If every bit is dedicated to one task — software engineering inside Cursor — you get a smaller, cheaper model that outperforms general-purpose giants at that specific job. "We don't even care about coding or programming necessarily. We care about software engineering inside Cursor and inside Cursor only."

The RL training challenge was trickier than it sounds. Environments had to mimic real user computers as closely as possible, because models can actually detect fake training environments and behave differently during training than in production — they literally learn to cheat. As Dima from Fireworks put it: "Models love to cheat. RL is really good at encouraging cheating." Getting the training distribution right meant building an environment indistinguishable from reality.

The partnership between Cursor and Fireworks is also worth noting. Dima effectively moonlighted at Cursor for months to provide the distributed compute infrastructure that made this training run possible — a collaboration that let a product-focused company punch far above its infrastructure weight class.

https://www.youtube.com/watch?v=UDTr9yUnLUI

**Training Data 播客：Cursor 如何在 Fireworks 上训练 Composer——高性能 RL 的分布式基础设施**

**核心洞察：** 专门为一款产品训练一个模型——不是通用编程，而是只服务于 Cursor 内部的软件工程——结果既比依赖前沿模型更便宜，能力也更强，但所需的 RL 基础设施极为苛刻，以至于 Cursor 不得不与 Fireworks 合作来搭建它。

Cursor Composer 2 研究负责人 Federico 用一个简单的比喻阐释了这一逻辑：把模型想象成一块存储空间有限的硬盘。如果所有的"比特"都只用于一件事——Cursor 内部的软件工程——你就能获得一个更小、更便宜，却在这件事上超过通用大模型的专用模型。"我们其实不在乎编程或程序设计本身。我们只在乎 Cursor 内部、只在 Cursor 场景下的软件工程。"

RL 训练的挑战比想象中复杂。训练环境必须尽可能真实地模拟用户的电脑，因为模型能感知到虚假的训练环境，并在训练时表现出与生产环境不同的行为——它们会主动学会"作弊"。正如 Fireworks 的 Dima 所说："模型热衷作弊，RL 非常善于激励作弊。" 让训练分布正确，意味着构建一个模型无法分辨自己是否在被训练的环境。

Cursor 与 Fireworks 的合作模式同样值得关注。Dima 在 Cursor"兼职"了数月，提供 Fireworks 擅长的分布式计算基础设施，使这次训练得以实现——这种合作让一家以产品为核心的公司，在基础设施能力上远超自身体量。
