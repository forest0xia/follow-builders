**Dan Shipper** (CEO, Every)

Dan Shipper pushed back on a pattern he finds intellectually dishonest: "we got a tool to perform poorly" as a form of science or journalism. **Anthropic Engineering: An update on recent Claude Code quality reports**
https://www.anthropic.com/engineering/april-23-postmortem

Anthropic published a detailed postmortem on reports of degraded Claude Code quality over the past two months. The root cause turned out to be three separate, unrelated bugs that happened to overlap in time — making the aggregate effect look like broad, inconsistent regression across all users.

The first: on March 4, Claude Code's default reasoning effort was quietly lowered from "high" to "medium" to reduce UI-freezing latency for users on long thinking tasks. Users noticed lower intelligence and complained; Anthropic reversed the change on April 7. All users now default to xhigh effort on Opus 4.7 and high on all other models.

The second: a March 26 caching optimization introduced a bug that caused Claude to drop its prior reasoning on *every turn* of a session after it had been idle for over an hour, instead of just once. Claude would keep executing tasks but increasingly without memory of its own prior reasoning — manifesting as forgetfulness, repetition, and odd tool choices. A side effect: repeated cache misses drained usage limits faster than expected. Fixed April 10.

The third: on April 16, a system prompt addition ("keep text between tool calls to ≤25 words, final responses to ≤100 words") was shipped alongside Opus 4.7 to reduce verbosity. It was internally tested for weeks with no regressions — but caused a measurable coding quality drop at scale. Reverted April 20.

As of April 23, Anthropic is resetting usage limits for all subscribers and rolling out improved safeguards: broader per-model evals for every system prompt change, tighter controls on prompt modifications, and more internal staff on the exact public build.

Anthropic 发布了一份详细的事故复盘报告，针对过去两个月 Claude Code 质量下降的用户反馈进行说明。根本原因是三个相互独立的 bug 在时间上恰好重叠，叠加效果看起来像是对所有用户的大范围、不一致的性能退化。

第一个问题：3 月 4 日，Claude Code 的默认推理强度从"高"被悄悄调低至"中"，以减少长思考任务中导致 UI 冻结的延迟。用户发现智能水平下降并反映，Anthropic 于 4 月 7 日撤销了这一改动。目前所有用户在 Opus 4.7 上默认为 xhigh 强度，其他所有模型默认为 high。

第二个问题：3 月 26 日的一次缓存优化引入了一个 bug，导致 Claude 在会话空闲超过一小时后，每一轮都会丢弃之前的推理记录，而非仅执行一次清理。Claude 会继续执行任务，但逐渐失去对自身先前推理的记忆——表现为健忘、重复和奇怪的工具选择。副作用是：反复的缓存未命中导致用量配额消耗速度超出预期。已于 4 月 10 日修复。

第三个问题：4 月 16 日，随 Opus 4.7 一并上线了一条系统提示补充（"工具调用之间的文字不超过 25 词，最终回复不超过 100 词"），旨在减少冗余输出。内部测试数周未见回归，但大规模上线后导致编程质量明显下降。已于 4 月 20 日回滚。

自 4 月 23 日起，Anthropic 重置了所有订阅用户的使用配额，并推出改进后的安全保障措施：对每次系统提示变更进行更全面的分模型评估、对提示词修改实施更严格的管控，并让更多内部员工使用与公众完全相同的正式版本。

**Claude Blog: Built-in memory for Claude Managed Agents**
https://claude.com/blog/claude-managed-agents-memory

Claude Managed Agents now has built-in memory in public beta. Agents can learn from every session, with memories stored as files — giving developers full export, API management, and control over what agents retain. The memory layer mounts directly onto the filesystem, so agents use the same bash and code-execution capabilities they already rely on, rather than a separate retrieval system.

For production deployments, memory supports scoped permissions, audit logs, and shared stores across multiple agents. Different agents can have different access (e.g., an org-wide read-only store alongside per-user read-write stores). Multiple agents can work against the same store concurrently without conflicts, and all changes are tracked so you can trace exactly which agent and session a memory came from — or roll back any change.

Early results from teams using it: Rakuten's long-running agents cut first-pass errors by 97% using cross-session memory; Wisedocs sped up document verification by 30%; Netflix agents carry context including mid-conversation human corrections across sessions without manual prompt updates.

Claude Managed Agents 的内置记忆功能现已进入公开测试阶段。Agent 可以从每次会话中学习，记忆以文件形式存储——开发者可完整导出、通过 API 管理，并完全掌控 agent 的记忆内容。记忆层直接挂载到文件系统，agent 使用的是其已经熟悉的 bash 和代码执行能力，而非另一套独立的检索系统。

面向生产环境部署，记忆功能支持权限范围控制、审计日志以及多 agent 共享存储。不同 agent 可拥有不同访问权限（例如组织级只读存储与用户级读写存储并存）。多个 agent 可同时操作同一存储而不产生冲突，所有变更均有记录可追溯——精确到哪个 agent、哪次会话产生了哪条记忆——并支持回滚任意历史版本。

已在早期使用团队中取得的成果：Rakuten 的长期运行 agent 通过跨会话记忆将首次通过错误率降低了 97%；Wisedocs 的文件核验速度提升了 30%；Netflix 的 agent 能够跨会话携带上下文（包括人类在对话中途的纠正），无需手动更新 prompt。
https://claude.com/blog/claude-managed-agents-memory

**Training Data: ElevenLabs' Mati Staniszewski — How Voice Becomes the Interface for Everything**
https://www.youtube.com/playlist?list=PLOhHNjZItNnMm5tdW61JpnyxeYH5NDDx8

**The Takeaway:** Voice isn't just another AI modality — it's the primary interface to the physical and ambient computing world, and ElevenLabs is building the full audio stack to own that transition.

ElevenLabs CEO Mati Staniszewski traces the company's founding to a culturally specific frustration: in Poland, all foreign films are dubbed by a single monotone narrator regardless of gender or emotion. That absurdity crystallized a vision — everyone should be able to speak any language with the same emotion and intonation as their own voice. From that kernel, ElevenLabs built outward into the full audio domain: text-to-speech, speech-to-text, dubbing, voice agents, and music.

What's counterintuitive about ElevenLabs: they monetized early and deliberately, treating revenue as a path to model independence rather than raising massive capital upfront. Staniszewski attributes much of their early edge to hiring strategy — scraping GitHub for audio researchers by their work rather than their location, assembling a remote team between London and Warsaw at a time when the space had almost no researchers.

The most striking insight from the conversation is about voice agents and overlooked markets. Beyond the obvious (customer support), Staniszewski sees the biggest opportunity in citizen services: government access, healthcare, and education. He cites Ukraine's government voice agent — deployed during wartime so citizens can call in to get frontline information, educational content, and safety guidance. He describes a future where you can learn physics with Richard Feynman or negotiate with Chris Voss in real time through a voice agent — citing Masterclass's interactive Gordon Ramsay cooking coach as a live example.

On building ElevenLabs at 400+ people and $400M+ ARR: every team — including legal, people, and go-to-market — has an embedded engineer. With "vibe coding" becoming ubiquitous, non-technical teams need technical reviewers, especially for outputs with security implications. ElevenLabs keeps teams under 10 people and has no titles, optimizing entirely for impact.

On moats: Staniszewski distinguishes between speech-to-text (technology, with Jensen Huang's blessing) and text-to-speech (artistry). Emotional intelligence — the ability to detect and match a user's emotional state in real time — is the next step change they're working toward internally. Agent-to-agent voice communication is already emerging: at a hackathon 18 months ago, two ElevenLabs voice agents detected each other as non-human and spontaneously switched to a more efficient transmission protocol.

对话中最令人印象深刻的洞察来自语音 agent 和被忽视的市场。除了显而易见的客服场景，Staniszewski 认为最大的机会在于公民服务：政府服务、医疗健康和教育。他以乌克兰政府语音 agent 为例——该系统在战时部署，让公民可以拨打电话获取前线信息、教育内容和安全指引。他描绘了一个未来图景：你可以通过语音 agent 与 Richard Feynman 实时学物理，或与 Chris Voss 练习谈判——并以 Masterclass 上线的互动式 Gordon Ramsay 烹饪教练作为已落地的案例。

关于在 400+ 人规模、年收入超 4 亿美元的情况下运营 ElevenLabs：每个团队——包括法务、人力和市场——都有内嵌工程师。随着"氛围编程"日趋普遍，非技术团队也需要技术审核者，尤其是涉及安全影响的输出内容。ElevenLabs 将团队规模控制在 10 人以下，且不设职级，一切以影响力为导向。

关于护城河：Staniszewski 区分了语音转文字（技术，获得了 Jensen Huang 的认可）和文字转语音（艺术性）。情感智能——即实时感知并匹配用户情绪状态的能力——是他们内部正在攻克的下一个重大突破。而 agent 之间的语音通信已初露端倪：在 18 个月前的一次黑客松中，两个 ElevenLabs 语音 agent 相互识别出对方是非人类，并自发切换到一种更高效的信息传输协议。
