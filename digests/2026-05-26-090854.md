### The MAD Podcast with Matt Turck: Why AWS and Azure Cannot Run Autonomous AI – Ivan Burazin (Daytona)

**The Takeaway:** AWS and Azure were architected for stateless, ephemeral workloads — and that fundamental design choice is exactly why they can't serve as the backbone for autonomous AI agents, which need persistent, stateful computers of their own.

Ivan Burazin is the CEO of Daytona, a startup building sandbox infrastructure for AI agents. He comes with sixteen years in the developer tools space — he was building cloud IDEs back in 2009, before Docker and Kubernetes even existed — so when he talks about what the current agent infrastructure wave is missing, he's speaking from hard-won experience.

Burazin's central argument is deceptively simple: agents are digital knowledge workers, and knowledge workers need computers. Not a tool call, not a Lambda function that spins up and dies — a persistent machine with a filesystem, a browser, the ability to run code, and its own identity. The analogy he keeps returning to is his own board meeting workflow: he asked Claude to pull bank data, and Claude said "just log in and give me access." His reaction — "I will not give you access" — crystallized why running agents on your own machine is a security nightmare. The solution is giving the agent its own isolated environment, complete with its own phone number for 2FA, its own accounts with scoped permissions, and a hard kill switch if something goes wrong.

The reason hyperscalers can't simply offer this is structural: "The way the chassis is built, the engine, the weight balance — it's completely different. You can do a lot of things with both, but they're not fundamentally the same thing." Cloud infrastructure was optimized for stateless web apps where you never want state to change unexpectedly. Agents need the opposite: persistent, stateful machines that keep running whether you're on your laptop, your phone, or asleep.

Daytona ended up throwing out Kubernetes and building their own scheduler because none of the existing options were designed for fast-starting, stateful, long-running machines. "None of these were made for these super fast, stateful, long-running machines. So we're like, we have to do this again." The result gives them the ability to offer four different isolation models and spin up sandboxes fast enough that GPU utilization isn't wasted waiting on CPU compute — a distinction that matters enormously for RL researchers running thousands of sandboxes in parallel.

Burazin also flags an underappreciated bottleneck: a global CPU shortage may be coming. Sandboxes are CPU-heavy by nature, and as agent workloads scale, the demand for plain compute — not GPUs — could get very tight, very fast.

**The MAD Podcast with Matt Turck: Why AWS and Azure Cannot Run Autonomous AI – Ivan Burazin (Daytona)**

**核心观点：** AWS 和 Azure 的底层架构是为无状态、短暂性工作负载设计的 — 而正是这一根本性的设计选择，决定了它们无法成为自主 AI agent 的基础设施，因为 agent 需要的是持久化、有状态的专属计算机。

Ivan Burazin 是 Daytona 的 CEO，这家公司专为 AI agent 构建 sandbox 基础设施。他在开发者工具领域深耕十六年，早在 2009 年 Docker 和 Kubernetes 诞生之前就在构建云端 IDE，因此当他谈及当前 agent 基础设施缺什么时，每一句话背后都有真实经历撑腰。

Burazin 的核心论断简洁有力：agent 就是数字知识工作者，知识工作者需要电脑。不是一次 tool call，不是一个运行完就销毁的 Lambda 函数 — 而是一台有文件系统、有浏览器、能运行代码、有自己身份的持久化机器。他反复用自己做董事会报告的经历来打比方：他让 Claude 去银行拉数据，Claude 说"你登录一下给我访问权限"。他的反应 — "我不会给你访问权限的" — 让他彻底想清楚了为什么把 agent 跑在自己电脑上是个安全噩梦。解决方案是给 agent 一个隔离的独立环境，有自己的手机号用于双因素认证，有权限受限的专属账号，以及必要时可以直接拔插头的强制终止机制。

超大云服务商无法简单提供这种能力，原因是结构性的："底盘的构造方式、引擎、重量分布都完全不同。两者能做很多事，但根本上不是同一种东西。"云基础设施为无状态 web 应用做了优化 — 你永远不希望状态意外改变。而 agent 恰恰需要相反的东西：无论你用的是笔记本、手机还是已经睡觉了，机器都要持续运行、保持状态。

Daytona 最终放弃了 Kubernetes，自己从头写了调度器，因为现有的所有方案都不是为快速启动、有状态、长时间运行的机器设计的。"这些东西没有一个是为这种超快速、有状态、长时间运行的机器而设计的。所以我们说，得重新做一遍。"新调度器让他们可以提供四种不同的隔离模型，并且启动 sandbox 足够快，不会因为 CPU 等待时间拖累 GPU 利用率 — 这对于并行跑成千上万个 sandbox 做强化学习的研究团队来说，差别极大。

Burazin 还点出了一个被低估的瓶颈：全球 CPU 短缺可能即将到来。Sandbox 本质上是 CPU 密集型的，随着 agent 工作负载规模扩大，对普通算力（而非 GPU）的需求可能会迅速趋紧。
