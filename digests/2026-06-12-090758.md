**Anthropic Engineering: How we contain Claude across products**
https://www.anthropic.com/engineering/how-we-contain-claude

**The Takeaway:** As Claude agents gain real-world access, Anthropic has learned that containment at the environment layer — not just model-level safety prompts — is the only defense that holds when everything else fails.

Anthropic Engineering published a detailed post on how they architect agent containment across three Claude platforms: claude.ai, Claude Code, and Claude Cowork. Twelve months ago, granting Claude access sufficient to take down an internal Anthropic service would have been unthinkable; today it's routine. The central insight: as agent capabilities grow, the blast radius of any failure also grows, so the engineering question shifts from "should we deploy?" to "how do we hard-cap what can go wrong?"

The post describes three isolation patterns. **claude.ai** uses ephemeral gVisor containers (server-side only, no local filesystem access). **Claude Code** uses a human-in-the-loop sandbox (Seatbelt on macOS, bubblewrap on Linux), which reduced permission prompts by 84% — but the team found that approval fatigue meant users were rubber-stamping 93% of prompts, defeating the purpose. The new Claude Code auto mode uses a model-based classifier to automate safe approvals. **Claude Cowork** runs inside a local VM (Apple's Virtualization framework / HCS on Windows), the strongest isolation, designed for non-technical users who can't evaluate bash commands.

Three hard-won lessons from real incidents stand out. First, a **pre-trust execution vulnerability**: a malicious repository could include `.claude/settings.json` that ran hooks *before* the "do you trust this folder?" prompt even appeared — fix was to defer all project config parsing until after user consent. Second, a **phishing vector via the user**: in a red team, an employee was tricked into pasting a prompt that exfiltrated `~/.aws/credentials` — the model completed it 24 out of 25 tries, because the request looked like a legitimate user instruction. Only egress controls stopped it, not the model. Third, an **allowlist-as-capability-grant failure**: Claude Cowork allowed `api.anthropic.com`, which an attacker abused to exfiltrate workspace files to their own Anthropic account via the Files API — fixed with an in-VM MITM proxy that only passes requests with the VM's own session token.

The core principle the team keeps returning to: "Design for containment at the environment layer first, then steer behavior at the model layer." Probabilistic defenses (classifiers, system prompts) can never be 100%; deterministic boundaries (VMs, egress controls) are what hold when everything probabilistic misses.

**Anthropic Engineering：我们如何在各产品中对 Claude 进行沙箱隔离**
https://www.anthropic.com/engineering/how-we-contain-claude

**核心结论：** 随着 Claude agent 获得真实世界的操作权限，Anthropic 发现：当所有其他防御手段失效时，唯一真正可靠的是**环境层的隔离**，而非仅依赖模型层的安全提示。

Anthropic Engineering 发布了一篇深度技术文章，详述他们如何在三个 Claude 平台（claude.ai、Claude Code 和 Claude Cowork）中设计 agent 隔离架构。一年前，给予 Claude 足以破坏 Anthropic 内部服务的权限还是不可想象的；而现在这已成为常态。核心洞察在于：随着 agent 能力增强，任何失败的影响范围也在扩大，工程问题从"是否部署"转变为"如何硬性限制最坏情况"。

文章描述了三种隔离模式。**claude.ai** 使用临时 gVisor 容器（纯服务器端，无本地文件系统访问）。**Claude Code** 使用"人在回路"沙箱（macOS 上的 Seatbelt，Linux 上的 bubblewrap），将权限提示减少了 84%——但团队发现"审批疲劳"导致用户对 93% 的提示选择允许，效果适得其反。新的 Claude Code 自动模式使用基于模型的分类器来自动处理安全审批。**Claude Cowork** 运行在本地 VM 中（macOS 用苹果虚拟化框架，Windows 用 HCS），提供最强隔离，专为无法判断 bash 命令的非技术用户设计。

三个来自真实事故的重要教训：其一，**信任前代码执行漏洞**：恶意仓库可以在".claude/settings.json"中内嵌 hook，在"是否信任此文件夹"提示出现*之前*就已执行——修复方案是将所有项目配置解析推迟到用户确认信任之后。其二，**以用户为媒介的注入攻击**：红队演练中，一名员工被诱导粘贴了一段提示词，导致`~/.aws/credentials`被窃取——模型在 25 次尝试中成功执行了 24 次，因为请求看起来像合法用户指令，只有出站流量控制而非模型本身阻止了它。其三，**白名单即能力授予的失效**：Claude Cowork 将`api.anthropic.com`列为允许域名，攻击者利用 Files API 将工作区文件上传到自己的 Anthropic 账户——修复方案是在 VM 内部署中间人代理，只放行携带 VM 自身会话 token 的请求。

团队反复回归的核心原则："先在环境层设计隔离，再在模型层引导行为。"概率性防御（分类器、系统提示）永远无法达到 100%；确定性边界（VM、出站控制）才是所有概率性手段失效时的最后防线。

**Training Data — "Google DeepMind's Logan Kilpatrick: Why the Model Eats the Harness"**
https://www.youtube.com/watch?v=cMAs8z2dehs

**The Takeaway:** Logan Kilpatrick argues that what we call "the model" has already expanded far beyond a set of weights — it's an ever-growing system of scaffolding, and within 12 months much of what today's builders call "the harness" will simply be absorbed into the model itself.

Logan Kilpatrick runs Google AI Studio and the Gemini API, putting him at the center of how Google is shipping agentic products to developers. Fresh off Google I/O, he offered a rare insider look at the thinking behind Google DeepMind's agentic era.

His central thesis: the distinction between "model" and "harness" is dissolving. "Two years ago, when LLMs were popular, the model was actually just a set of weights... but we've just progressively, step by step, built an entire expanding sprawling system around the weights." Today's alpha — building your own agent harness — will be table stakes within a year, because the model will have absorbed it. He frames this not as a threat to builders, but as a natural progression: scaffolding leads the model's capabilities, the model digests it, then the alpha moves somewhere else.

On Google's agentic strategy: the company's internal agent harness, Antigravity, is now the through-line across all Google products — the same way Gemini became the through-line before it. Products from Search to Gmail to the Gemini app are all rebasing to Antigravity. And counterintuitively, Kilpatrick believes Google's agentic rollout has been *positive-sum* for search: "People are searching more. Agents are searching too."

On why Google hasn't won the developer coding market yet: "It's actually really hard to make a great coding model for this developer use case of really long running, suite work if you don't actually have a product that does that." That's why the Windsurf acquisition happened and why the team building Antigravity is what he calls internally "the Avengers of AI."

On DeepMind's culture: Kilpatrick describes it as a rare blend of deep research DNA (shaped by Demis Hassabis's Nobel Prize-winning scientific roots) and applied product work at billion-user scale. His favorite internal framing: "We can't let other people make the world a better place more than we can" — the Silicon Valley satire that, when you live inside it, starts to sound earnest.

On world models and Gemini Omni: Google's new Omni model is a single unified model (not routing between separate models) that can take any input and create any output. Its first major use case is AI-powered video editing, exemplified by a live demo where a photo was edited in real time to place a dog on stage during a talk — with other guests convincingly reacting to it. Kilpatrick's vision: Omni should amplify authentic human content, not replace it. "Our words can stay the same, and you can change these bits that are not personal and do something more interesting with them."

**核心结论：** Logan Kilpatrick 认为，我们所说的"模型"早已远不止一组权重——它是一个不断扩张的系统；而在未来 12 个月内，今天的开发者所称的"harness"大部分将被模型本身所吸收。
