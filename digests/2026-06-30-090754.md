### Anthropic Engineering

**How we contain Claude across products**

Anthropic Engineering breaks down how it contains agents across claude.ai, Claude Code, and Claude Cowork, each using a different isolation pattern matched to how much oversight users can realistically provide. claude.ai runs code in an ephemeral gVisor container; Claude Code relies on a human-in-the-loop sandbox (Seatbelt/bubblewrap) that cut permission prompts 84%; Claude Cowork runs inside a full VM with strict filesystem and egress controls. The piece also confesses to real misses: a vulnerability let project-local hooks execute before the "trust this folder" prompt appeared; a phishing red-team exercise got Claude to exfiltrate AWS credentials in 24 of 25 retries because the malicious instructions came from the user, not a tool; and a third-party disclosure showed an attacker-supplied API key could ride through the api.anthropic.com allowlist to upload files to the attacker's own account, fixed with an in-VM man-in-the-middle proxy that validates session tokens. On Gray Swan's Agent Red Teaming benchmark, Claude Opus 4.7 holds prompt-injection attack success to roughly 0.1% on a single attempt and roughly 5-6% after 100 adaptive attempts. The core lesson: "the weakest layer is the one you built yourself" — battle-tested sandboxes and hypervisors held up; Anthropic's own custom proxies were what broke.

**An update on recent Claude Code quality reports**

Anthropic Engineering published a postmortem on weeks of user reports that Claude Code felt "worse." The investigation traced the perceived degradation to three separate, unrelated bugs — the API itself was never affected. First, a March 4 change defaulted Claude Code's reasoning effort from high to medium to cut latency; it backfired and was reverted April 7, with Opus 4.7 now defaulting to xhigh and other models to high. Second, a March 26 caching optimization meant to clear stale thinking once after an hour of idle time instead cleared it on every subsequent turn, making Claude appear forgetful and repetitive while also causing extra cache misses that drained usage limits faster — fixed April 10. Third, an April 16 system-prompt tweak capping responses to roughly 100 words to reduce Opus 4.7's verbosity caused a 3% eval drop and was reverted April 20. All three issues were resolved by v2.1.116, and Anthropic reset usage limits for all subscribers. Notably, Opus 4.7 found the caching bug in a back-tested code review where Opus 4.6 had missed it. As the team put it: "This isn't the experience users should expect from Claude Code."

**Scaling Managed Agents: Decoupling the brain from the hands**

Anthropic Engineering details the architecture behind Claude Managed Agents, a hosted service for running long-horizon agents that separates the "brain" (Claude plus its harness) from the "hands" (sandboxes that execute code) and the "session" (a durable, append-only event log). Early on, bundling everything into one container created a classic infrastructure "pet" problem — a single failed container meant a lost session, and debugging meant shelling into a box that held user data. The fix: the harness now treats the sandbox as just another tool via execute(name, input) → string, so a dead container is just a retryable tool error, and a crashed harness can resume from the session log with wake(sessionId). This also closed a security gap — credentials are kept out of the sandbox entirely, routed through git remotes or an MCP proxy that pulls tokens from a vault. The payoff was concrete: median time-to-first-token dropped roughly 60%, and p95 fell over 90%, since containers are now provisioned only when actually needed. The framing: "Don't adopt a pet" — make every component replaceable so harnesses, sandboxes, and models can evolve independently as Claude gets smarter.

https://www.anthropic.com/engineering/managed-agents

Anthropic Engineering 详细介绍了 Claude Managed Agents 的架构设计，这是一项面向长周期 agent 任务的托管服务，将"大脑"（Claude 及其 harness）、"双手"（负责执行代码的沙箱）和"会话"（持久化的、仅追加的事件日志）三者解耦。早期把所有组件塞进同一个容器，带来了典型的基础设施"宠物问题"——一旦某个容器挂掉，整个会话就丢失了，而调试往往意味着要登录进一个还存有用户数据的机器里排查。解决方案是：让 harness 把沙箱当作普通工具调用，即 execute(name, input) → string，这样容器挂掉只是一次可重试的工具调用错误，而 harness 本身崩溃后也能通过 wake(sessionId) 从会话日志中恢复。这一改动同时补上了一个安全漏洞——凭证完全不进入沙箱，而是通过 git remote 或从密钥库读取 token 的 MCP 代理来传递。效果立竿见影：p50 首字延迟（TTFT）下降约 60%，p95 更是下降超过 90%，因为容器现在只在真正需要时才会被创建。团队的理念是："不要养'宠物'"——让每个组件都可被替换，这样 harness、沙箱和模型才能随着 Claude 变得更聪明而独立演进。

### Claude Blog

**New in Claude Managed Agents: self-hosted sandboxes and MCP tunnels**

Claude Blog announced that Claude Managed Agents can now run tool execution inside a sandbox you control and connect to private MCP servers inside your network, keeping both the execution environment and the services it touches within your enterprise's existing security perimeter. Self-hosted sandboxes (public beta) work with your own infrastructure or managed providers — Cloudflare (microVMs with zero-trust secret injection), Daytona (long-running, pausable/resumable sandboxes accessible over SSH), Modal (sub-second startup, scaling to hundreds of thousands of concurrent sandboxes), or Vercel (VPC peering with credentials injected at the network boundary so they never enter the sandbox). The agent loop itself — orchestration, context management, error recovery — still runs on Anthropic's infrastructure, but files, repos, and compute stay on your side. MCP tunnels (research preview, request access required) let agents reach internal databases, APIs, and ticketing systems through a lightweight gateway that makes a single outbound connection, with no inbound firewall rules or public endpoints needed. Early adopters cited include Amplitude (an internal design agent on Cloudflare), Clay (a GTM engineering agent on Daytona), and Rogo (an institutional-finance analyst agent on Vercel Sandbox).

https://claude.com/blog/claude-managed-agents-updates
