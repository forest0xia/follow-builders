### Anthropic Engineering: Scaling Managed Agents — Decoupling the Brain from the Hands

Anthropic's engineering team published a deep technical post on the architecture behind Managed Agents, their hosted service for running long-horizon AI agents at scale. The central insight is structural: the harness that drives Claude should not live in the same container as the tools Claude uses. By separating the "brain" (Claude and its harness) from the "hands" (sandboxes, execution environments) and the "session" (the persistent log of everything that happened), each component can fail, scale, or be replaced independently.

Anthropic Engineering：《扩展 Managed Agents：将大脑与双手解耦》

Anthropic 工程团队发布了一篇关于 Managed Agents 架构的深度技术文章，介绍了他们用于大规模运行长周期 AI agent 的托管服务。核心洞见在于结构设计：驱动 Claude 的 harness 不应与 Claude 所使用的工具共处同一容器。通过将"大脑"（Claude 及其 harness）与"双手"（沙箱、执行环境）以及"会话"（所有发生事件的持久化日志）分离，每个组件都可以独立地故障、扩展或被替换。

The old approach — everything in one container — created what engineers call a "pet": a hand-tended system you can't afford to lose. If the container crashed, the session was gone. Debugging required opening a shell into a container that also held user data, making meaningful observability nearly impossible. The new design treats containers as "cattle": the harness calls sandboxes the same way it calls any other tool, `execute(name, input) → string`. If a container dies, the harness catches it as a tool-call error and Claude can simply retry with a fresh one.

旧有方案——所有内容放在一个容器中——制造了工程师所说的"宠物"：一个你不敢失去的、需要精心呵护的系统。一旦容器崩溃，会话就消失了。调试需要进入一个同时存有用户数据的容器，实质上的可观测性几乎无从谈起。新设计将容器视为"牲口"：harness 调用沙箱的方式与调用任何其他工具一样，`execute(name, input) → string`。如果容器宕机，harness 将其作为工具调用错误捕获，Claude 可以直接用新容器重试。

The security implications are equally important. In the old coupled design, any prompt injection only had to convince Claude to read its own environment to get credentials — Claude's generated code ran in the same container as auth tokens. The fix: credentials are never reachable from the sandbox. For Git repos, access tokens are used during initialization and wired into the local remote so Claude can push and pull without ever handling the token. For custom tools, OAuth tokens live in a secure vault and are only accessed via a dedicated MCP proxy. As the post notes: *"The harness is never made aware of any credentials."*

安全影响同样至关重要。在旧的耦合设计中，prompt 注入只需说服 Claude 读取自身环境就能获取凭证——Claude 生成的代码与 auth token 运行在同一容器中。解决方案：凭证永远无法从沙箱访问。对于 Git 仓库，访问 token 在初始化时使用并接入本地 remote，Claude 无需直接接触 token 就能执行 push 和 pull。对于自定义工具，OAuth token 存储在安全保险库中，仅通过专用 MCP 代理访问。正如文章所写：*"harness 对任何凭证一无所知。"*

The performance numbers make the architectural argument concrete. Previously, every session paid the full container setup cost upfront — even sessions that never touched the sandbox. After decoupling, containers are provisioned on demand via tool call, and inference begins as soon as the orchestration layer pulls pending events from the session log. **p50 time-to-first-token dropped roughly 60%; p95 dropped over 90%.**

The session log as an external, durable object also solves a long-standing problem in long-horizon agents: irreversible decisions about what to keep in context. Rather than compacting or trimming context and losing information permanently, the session log acts as a context object outside the model's window. The brain can call `getEvents()` to fetch positional slices of the event stream — rewinding a few events, or jumping to any point — and transform them before passing them to Claude. Context management is pushed into the harness, which can evolve as models improve, while the session remains a durable, interrogable record.

性能数据让架构论据变得具体。此前，每个会话都要预先支付完整的容器启动成本——即便是从不接触沙箱的会话也不例外。解耦之后，容器按需通过工具调用进行配置，推理在编排层从会话日志拉取待处理事件后即可开始。**p50 首 token 延迟下降约 60%；p95 下降超过 90%。**

会话日志作为外部持久化对象，也解决了长周期 agent 长期存在的一个难题：关于保留哪些上下文的不可逆决策。与其压缩或裁剪上下文而永久丢失信息，会话日志充当了存在于模型上下文窗口之外的上下文对象。大脑可以调用 `getEvents()` 获取事件流的位置切片——回溯几个事件，或跳转到任意时间点——并在传递给 Claude 之前进行转换。上下文管理被推入 harness，随着模型改进而演进，而会话则保持为持久、可查询的记录。

### No Priors: Baseten CEO Tuhin Srivastava on the AI Inference Crunch, Custom Models, and Building the Inference Cloud

**The Takeaway:** 95% of tokens served on Baseten's inference cloud come from customized models — virtually no one is running vanilla open-source weights — and the compute crunch is far worse than the public narrative suggests.

**要点：** Baseten 推理云上 95% 的 token 来自定制化模型——几乎没有人在运行原始的开源权重——而算力短缺比公开叙事所描述的严峻得多。

Tuhin Srivastava founded Baseten as an inference cloud and has grown it 30x in roughly a year, with revenue on track to cross a billion dollars in 2026. The No Priors hosts put two recurring questions to him: does the independent application layer get to exist alongside the frontier labs, and how bad is the GPU supply crunch really?

On the first question, Srivastava's answer is structural rather than sentimental. Companies survive if their value is encoded in workflows rather than in models. His example: Abridge, an ambient scribe used across US hospitals, has built deep integrations into clinician workflows and EMR systems. A frontier model lab simply doesn't have access to that user signal. *"To the extent that [value] is encoded in workflows, that is where they will be able to develop moat."* He adds that by serving AI-native companies like Abridge, Cursor, and Open Evidence, Baseten effectively gets a translation of enterprise requirements a cycle ahead — what data retention policies health systems will eventually demand, what model transparency requirements look like at scale.

Tuhin Srivastava 创立 Baseten 作为推理云，在约一年内实现了 30 倍增长，2026 年营收有望突破十亿美元。No Priors 主持人向他提出了两个反复出现的问题：独立应用层能否与前沿实验室并存，以及 GPU 供应短缺究竟有多严重？

关于第一个问题，Srivastava 的回答是结构性的，而非情感性的。企业的生存之道在于将价值编码在工作流中，而非模型中。他举例：Abridge 是一款被美国各大医院广泛使用的 ambient 医疗记录工具，已与临床工作流和 EMR 系统深度集成。前沿模型实验室根本无法获取这种用户信号。*"只要价值被编码在工作流中，那就是他们能够建立护城河的地方。"* 他补充说，通过服务 Abridge、Cursor 和 Open Evidence 等 AI 原生公司，Baseten 实际上提前一个周期获得了企业需求的转化——医疗系统最终会要求什么数据保留策略，大规模下的模型透明度要求是什么样的。

On the custom model trend: nearly all of Baseten's token volume is dedicated inference, and of that, 90–95% involves some customer modification to the base model — whether for quality (fine-tuning, RLHF), performance (quantization, compilation), or both. *"No one is just running the vanilla open source weights."* To accelerate this capability, Baseten acquired Parz, a post-training startup that was already a Baseten customer. The key insight from that work: inference and post-training are two sides of the same problem. How you train a model affects how you need to quantize it for inference; inference data can feed reward functions that drive the next round of post-training.

关于定制模型趋势：Baseten 几乎所有的 token 量都是专用推理，其中 90-95% 涉及客户对基础模型的某种修改——无论是为了质量（微调、RLHF）、性能（量化、编译）还是两者兼顾。*"没有人只是在运行原始的开源权重。"* 为了加速这一能力，Baseten 收购了 Parz，一家已是 Baseten 客户的后训练初创公司。这项工作的核心洞见：推理和后训练是同一问题的两面。模型的训练方式影响着推理时如何量化；推理数据可以反哺奖励函数，驱动下一轮后训练。

On supply: the crunch is worse than the headlines. Baseten runs clusters at "mid-nineties utilization" across 90 clusters in 18 clouds worldwide, and has a standing 4PM daily meeting just to manage capacity for current demand. Even when compute is nominally available, many newer cloud providers haven't run data centers before and don't understand inference SLAs — Srivastava estimates only about a dozen good clouds exist, with three or four in a "gold tier." His advice for builders on when to invest in custom models: prove product-market fit first with the best available model. *"No post-training pre-product-market-fit"* is his update to the old "no GPUs pre-PMF" meme. Once you have differentiated user signal, then optimize for better, faster, cheaper.

关于供应：短缺比标题新闻描述的更为严重。Baseten 在全球 18 个云上的 90 个集群中以"九十几的利用率"运行，并且专门设有每日下午 4 点的常规会议来管理当前需求的算力。即使算力名义上可用，许多新兴云服务商以前从未运营过数据中心，不了解推理 SLA——Srivastava 估计全球只有约十几个靠谱的云，其中三四个属于"黄金层"。他对构建者关于何时投资定制模型的建议：先用最佳可用模型证明产品市场契合度。*"没有产品市场契合度就别做后训练"*，这是他对旧有"融资前别囤 GPU"梗的更新。一旦拥有差异化的用户信号，再去优化更好、更快、更便宜。
