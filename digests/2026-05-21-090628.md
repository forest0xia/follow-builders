**AI & I by Every — Inside Stainless: The Developer Tools Startup Anthropic Just Bought for $300 Million**

**The Takeaway:** The future of AI agents is not hundreds of MCP tools — it is a code execution sandbox, where the model writes and runs code directly against an API's SDK.

Alex Rattray, the founder and CEO of Stainless, built the company to solve what sounds simple but is deceptively hard: getting computers to talk to each other cleanly. Stainless powers the SDKs and APIs for OpenAI, Anthropic, and most of the major tech companies you use. With Anthropic's reported $300M acquisition, the conversation about MCP servers that Every CEO Dan Shipper recorded back in October now reads like a technical thesis for the deal.

The core problem Rattray identifies: while everyone is excited about MCP (Model Context Protocol, the standard for connecting AI agents to external tools), the current implementations are badly broken. The vision — an AI that can navigate five different SaaS apps, refund a customer, send a discount code, and notify an account manager, all from one natural-language prompt — is real and achievable. But today's models can't handle the hundreds of MCP tool definitions required to get there; you'd blow through your entire context window just loading the Stripe API spec.

Rattray's design principles for MCP servers that actually work: keep the number of tools small, make each tool name and description laser-precise, minimize inputs and outputs, and strip unnecessary data from API responses. For large APIs, Stainless uses a "dynamic mode" with just three tools — list endpoints, inspect an endpoint, execute an endpoint — so context costs stay flat no matter how big the API is.

His long-term bet, however, is on code execution over tooling. "We haven't figured out how to expose an API ergonomically to an LLM in the same way that we've figured out how to expose it ergonomically to a Python developer," he says. His solution: give the model one tool — write TypeScript and run it — plus a doc search tool. The AI writes code against the SDK, runs it in a sandboxed server, gets back only the relevant output, and consults the docs when it hallucinates a wrong function name. Static type checking catches errors before they propagate. This approach nearly eliminates context overhead and runs dramatically faster than round-tripping through dozens of model calls.

Rattray also uses MCP for his own business operations at Stainless, connecting Notion, HubSpot, Gong, and a read-only Postgres copy so he can ask questions like "which customers signed up last week?" and get synthesized answers across all four systems in one shot.

**要点：** AI agent 的未来不是几百个 MCP 工具，而是一个代码执行沙箱——让模型直接针对 API 的 SDK 编写并运行代码。

Stainless 的创始人兼 CEO Alex Rattray 创办这家公司，是为了解决一个听起来简单、实则极为棘手的问题：让计算机之间的通信干净可靠。Stainless 为 OpenAI、Anthropic 以及大多数主流科技公司提供 SDK 和 API 的底层能力。随着 Anthropic 以约 3 亿美元收购 Stainless，Every CEO Dan Shipper 去年 10 月录制的这期关于 MCP server 的对话，如今读来已然是这笔收购背后的技术逻辑陈述。

Rattray 指出的核心问题是：尽管大家对 MCP（Model Context Protocol，AI agent 与外部工具的连接标准）充满期待，但目前的实现方式严重失效。那个愿景——AI 能同时穿梭于五个不同的 SaaS 工具，完成退款、发优惠码、通知客户经理，全程只需一句自然语言指令——在理论上是可行的，但当前的模型根本无法承载实现这一切所需的数百个 MCP 工具定义；光是加载 Stripe 的 API 规范，就足以耗尽你的全部上下文窗口。

Rattray 总结出的可用 MCP server 设计原则：工具数量要少，每个工具的名称和描述要精准，输入输出要精简，并过滤掉 API 响应中多余的数据。对于大型 API，Stainless 采用"dynamic mode"——只暴露三个工具：列出 endpoint、查看某个 endpoint、执行 endpoint——无论 API 有多大，上下文成本都保持恒定。

但他对长期的押注是代码执行，而非工具化。他说："我们还没有弄明白如何像为 Python 开发者设计一样，为 LLM 设计符合人体工学的 API 接口。"他的方案是：给模型一个工具——写 TypeScript 并执行它——加上一个文档检索工具。AI 针对 SDK 编写代码，在沙箱服务器中运行，只取回相关输出，遇到函数名幻觉时去查文档。静态类型检查会在错误传播前及时拦截。这一方案几乎消除了上下文开销，运行速度也远快于通过数十次模型调用来回跳转的方式。

Rattray 自己也在 Stainless 的日常运营中使用 MCP，将 Notion、HubSpot、Gong 和一个只读的 Postgres 副本接入同一个系统，这样只需一次提问，比如"上周有哪些新客户注册了？"就能跨四个系统得到一份综合答复。
