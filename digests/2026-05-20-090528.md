**Anthropic Engineering — "Scaling Managed Agents: Decoupling the brain from the hands"**

Anthropic Engineering published an architectural deep-dive into how they rebuilt Managed Agents by separating three core components: the session (a durable, append-only event log), the harness (the "brain" — Claude plus its control loop), and the sandbox (the "hands" — execution environment). The original monolithic design treated each container as a "pet" — irreplaceable, hard to debug, tightly coupled. The new architecture makes each component cattle: if a harness crashes, a new one wakes with wake(sessionId) and picks up from the last event. If a sandbox fails, it's treated as a tool-call error and a new one is provisioned automatically.

The performance results are striking: p50 time-to-first-token dropped roughly 60%, and p95 dropped over 90% — because containers are now only provisioned when actually needed, via a tool call. Security improved structurally too: generated code now runs in a sandbox that never touches credentials; tokens are either bundled with resources at initialization or held in a vault accessed only through a dedicated proxy.

https://www.anthropic.com/engineering/managed-agents

Anthropic Engineering 发布了关于 Managed Agents 架构重构的深度技术解析。他们将三个核心组件彻底解耦：session（持久化的只追加事件日志）、harness（"大脑"——Claude 及其控制循环）、sandbox（"双手"——执行环境）。原来的单体设计将每个容器视为"宠物"——不可替代、难以调试、高度耦合。新架构将每个组件都变成"牲口"：harness 崩溃后，新实例通过 wake(sessionId) 唤醒并从上一个事件继续；sandbox 失败时被视为工具调用错误，系统自动重新分配一个新实例。

性能提升效果显著：p50 首 token 时间（TTFT）降低约 60%，p95 降低超过 90%——因为容器现在只在真正需要时才通过工具调用来分配。安全性也从结构上得到改善：生成的代码现在运行在无法接触凭证的沙箱中；token 要么在初始化时与资源捆绑，要么存储在只能通过专用代理访问的安全保险库中。

https://www.anthropic.com/engineering/managed-agents

**Training Data — "Rebuilding IT From the Ground Up for the AI Age: Serval's Jake Stauch"**

**The Takeaway:** If building automation is easier than doing the task manually, people will always choose to automate — and that's the entire product thesis at Serval.

Jake Stauch, founder and CEO of Serval, is building what he calls "an AI-native ServiceNow": enterprise service management software that uses AI to generate and maintain the workflows and databases themselves, not just run on top of them. The core architecture uses a "cogen engine" that converts natural language descriptions into TypeScript workflows instantly. Once generated, those workflows run without re-generating code on every request — a key reason Serval's unit economics hold up well even at scale.

Stauch's most memorable line captures the whole bet: "If it were actually easier to build the automation, you would build the automation, because it's just, why wouldn't you?" This is in sharp contrast to traditional ServiceNow, where building a single workflow could take weeks of dedicated developer time — by which point the business process may have already changed.

The most contrarian take: Stauch argues the real moat for application-layer AI companies isn't the technology — it's customer intimacy. He's personally active in all 100+ customer Slack channels every day and sees it as their most defensible edge. "More and more of the moat for any startup is the customer insight, the empathy of actually understanding what they want. And if we can have a differentiated advantage around the customer insight, that's going to be much more valuable than having a product advantage which is copyable overnight."

On the "won't foundation models just do this?" question, his answer is focus: Anthropic added more ARR in recent months than ServiceNow has in 20 years — so why would they divert their best people into ITSM? Serval's structural answer is a two-agent architecture: an admin agent that builds and publishes approved tools, and a helpdesk agent that end users interact with, constrained to only use what the admins have sanctioned.

https://www.youtube.com/watch?v=j7ypvRUFY7M

**核心洞察：** 如果构建自动化比手动完成任务更简单，人们就会始终选择自动化——这正是 Serval 的核心产品理念。

Serval 创始人兼 CEO Jake Stauch 正在构建一个"AI 原生的 ServiceNow"：企业服务管理软件，利用 AI 生成并维护工作流和数据库本身，而不仅仅是运行在它们之上。核心架构使用"cogen 引擎"，将自然语言描述即时转化为 TypeScript 工作流。工作流一旦生成，每次执行时无需重新生成代码——这也是 Serval 即使在规模化场景下仍能保持良好单位经济模型的关键原因。

Stauch 最令人印象深刻的一句话道尽了整个产品的核心赌注："如果构建自动化实际上更容易，你就会去构建自动化，因为，为什么不呢？"这与传统 ServiceNow 形成了鲜明对比——在那里，构建一个工作流可能需要数周的专用开发时间，而那时业务流程可能早已发生了变化。

最具反差性的观点：Stauch 认为应用层 AI 公司真正的护城河不是技术，而是对客户的深度理解。他每天都亲自活跃在全部 100 多个客户 Slack 频道中，并将其视为最难被复制的竞争优势。"初创公司越来越多的护城河在于对客户的洞察，以及真正理解他们需求的同理心。如果我们能在客户洞察上建立差异化优势，这将比可以一夜之间被复制的产品优势更有价值。"

对于"基础模型不会把这一切都做掉吗？"这个经典问题，他的回答是专注：Anthropic 近期新增的 ARR 比 ServiceNow 二十年积累的还多——那他们为什么要把最优秀的人才转移到 ITSM 赛道呢？Serval 的结构性回答是双 agent 架构：一个 admin agent 负责构建和发布经过审批的工具，一个 helpdesk agent 供终端用户交互，且只能使用管理员明确授权的工具。
